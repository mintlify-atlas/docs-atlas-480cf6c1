## Live Preview

### Real-Time Rendering

The preview panel uses `react-markdown` to render your Markdown instantly as you type. The rendering is:

- **Secure**: No raw HTML injection (unless explicitly enabled via plugins)
- **Styled**: Uses Tailwind's typography plugin for beautiful defaults
- **Dark mode aware**: Automatically adjusts colors based on theme

### Synchronized Scrolling

Both editor and preview panels synchronize scroll positions:

<Info>
  Scroll in either panel, and the other follows proportionally. This keeps your current editing position visible in both raw and rendered views.
</Info>

Implementation detail from `app/page.tsx:130-138`:

```typescript
const handleScroll = (e: React.UIEvent<HTMLElement>) => {
  const source = e.currentTarget;
  const target = source === editorRef.current ? previewRef.current : editorRef.current;

  if (target) {
    const percentage = source.scrollTop / (source.scrollHeight - source.clientHeight);
    target.scrollTop = percentage * (target.scrollHeight - target.clientHeight);
  }
};
```

## Style Guide Selection

### Three Professional Guides

Choose from industry-standard style guides:

#### Google Developer Documentation Style Guide

Optimized for developer-facing content:

- Clear, concise language
- Active voice preference
- Consistent terminology
- Accessibility-first approach
- Extensive word list checks

**Best for**: API documentation, tutorials, developer guides

**Example rules**:
- Avoid phrases like "simply," "just," "easy"
- Use second person ("you") instead of "we" or "one"
- Prefer "to" over "in order to"
- Use em dashes (—) with no spaces

#### Microsoft Writing Style Guide

Focuses on customer-first, conversational content:

- Conversational tone
- Simple, everyday words
- Bias-free communication
- Consistent voice across products

**Best for**: User documentation, help content, product guides

**Example rules**:
- Avoid "please" in instructions
- Don't use "may" or "might" for permission
- Prefer "can" over "have the ability to"
- Use sentence case for headings

#### Red Hat Documentation Style Guide

Open source documentation standards:

- Precision and clarity
- Consistent terminology across docs
- Clear procedures and instructions
- Community-friendly tone

**Best for**: Open source projects, technical references, Linux documentation

**Example rules**:
- Use "you" sparingly (more objective tone)
- Prefer specific action verbs
- Consistent product name capitalization
- Proper use of technical terms

## Linting Engine

### Vale Integration

Verbalize uses [Vale](https://vale.sh/), the leading open-source prose linter:

- **Extensible**: Supports custom rules and style packages
- **Fast**: Processes typical documents in under 2 seconds
- **Accurate**: Line and column-level precision
- **Flexible**: Configurable severity levels and rule toggles

### How Linting Works

When you click "Run Linting," here's what happens server-side (from `app/api/lint/route.ts:45-150`):

<Steps>
  <Step title="Text Validation">
    ```typescript
    // Validate text length (max 1MB)
    if (text.length > MAX_TEXT_LENGTH) {
      return NextResponse.json({ error: 'Text exceeds maximum' }, { status: 400 });
    }
    
    // Validate style guide selection
    if (!ALLOWED_STYLES.includes(style)) {
      return NextResponse.json({ error: 'Invalid style guide' }, { status: 400 });
    }
    ```
  </Step>

  <Step title="Temporary File Creation">
    ```typescript
    // Create temp directory and file
    const tmpDir = await fs.mkdtemp(path.join(os.tmpdir(), 'vale-'));
    const tmpFile = path.join(tmpDir, 'doc.md');
    await fs.writeFile(tmpFile, text);
    ```
    
    Vale requires file-based input, so text is written to a temporary `.md` file.
  </Step>

  <Step title="Style Configuration">
    ```typescript
    // Create temporary styles directory
    const tempStylesDir = path.join(tmpDir, 'styles');
    
    // Symlink built-in styles (Google, Microsoft, RedHat)
    for (const styleFolder of existingStyles) {
      await fs.symlink(fullPath, path.join(tempStylesDir, styleFolder), 'dir');
    }
    
    // Add custom rules if provided
    if (customRules.length > 0) {
      const customStyleDir = path.join(tempStylesDir, 'Custom');
      // Write each custom rule .yml file
    }
    ```
  </Step>

  <Step title="Vale Execution">
    ```typescript
    // Generate dynamic .vale.ini
    const valeIni = `
    StylesPath = ${tempStylesDir}
    MinAlertLevel = suggestion
    [*.md]
    BasedOnStyles = ${style}, Custom
    `;
    
    // Execute Vale binary
    const { stdout } = await execFileAsync(valePath, [
      `--config=${iniFile}`,
      '--output=JSON',
      tmpFile
    ]);
    ```
  </Step>

  <Step title="Result Parsing">
    ```typescript
    // Parse JSON output
    const results = JSON.parse(stdout);
    
    // Return alerts for the temp file
    return NextResponse.json(results[tmpFile] || []);
    ```
  </Step>

  <Step title="Cleanup">
    ```typescript
    // Always clean up temp files
    finally {
      await fs.rm(tmpDir, { recursive: true, force: true });
    }
    ```
    
    All temporary files are removed immediately after linting completes.
  </Step>
</Steps>

### Alert Structure

Each Vale alert contains detailed information:

```typescript
interface ValeAlert {
  Action: {
    Name: string;           // Suggested action (e.g., "replace")
    Params: string[] | null; // Action parameters
  };
  Span: [number, number];   // Character positions [start, end]
  Check: string;            // Rule name (e.g., "Google.WordList")
  Description: string;      // Detailed explanation
  Link: string;             // External documentation URL
  Message: string;          // Short, actionable message
  Severity: string;         // "error" | "warning" | "suggestion"
  Match: string;            // The matched text
  Line: number;             // Line number (1-indexed)
}
```

### Severity Levels

Vale categorizes issues into three levels:

- **Error** (red badge): Critical issues that violate fundamental style rules
- **Warning** (yellow badge): Important issues that should be addressed
- **Suggestion** (blue badge): Optional improvements for better clarity or consistency

<Info>
  Verbalize sets `MinAlertLevel = suggestion` to show all three severity levels. You can modify this in the API route if you only want errors and warnings.
</Info>

## Custom Rules

### Uploading Custom Rules

Extend Verbalize with organization-specific Vale rules:

<Steps>
  <Step title="Create Vale Rule Files">
    Write `.yml` files following [Vale's rule syntax](https://vale.sh/docs/topics/styles/):

    ```yaml
    # company-terms.yml
    extends: substitution
    message: "Use '%s' instead of '%s'"
    level: error
    ignorecase: false
    swap:
      CompanyName: Acme Corp
      product-x: Acme Platform
    ```
  </Step>

  <Step title="Upload to Verbalize">
    1. Click the dashed upload box in **Custom Rules** section
    2. Select one or multiple `.yml` files (max 10 files)
    3. Each file appears in the custom rules list
  </Step>

  <Step title="Run Combined Linting">
    Click **"Run Linting"** to apply both base style guide AND custom rules together.

    The API dynamically creates a "Custom" style package and includes it in the Vale configuration.
  </Step>
</Steps>

### Custom Rule Limits

For security and performance:

- **Maximum rules**: 10 per session
- **Maximum rule size**: 10KB per file
- **File name sanitization**: Non-alphanumeric characters replaced with underscores
- **Path traversal prevention**: Only base filenames allowed

### Example Custom Rules

Click the "Example" link in the Custom Rules section to download a sample rule file. Here are more examples:

<CodeGroup>
```yaml company-terms.yml

  saas: SaaS

  back-end: backend
```

</CodeGroup>

```typescript
const handleAlertClick = (alert: ValeAlert, e: React.MouseEvent) => {
  // Calculate character offset

    offset += lines[i].length + 1;

  const start = offset + alert.Span[0] - 1;
  const end = offset + alert.Span[1];
```

## Performance & Limits

### Document Size Limits

For performance and security:

| Limit | Value | Reason |
|-------|-------|--------|
| **Maximum text length** | 1MB (1,048,576 bytes) | Prevents timeout on Vale execution |
| **Typical processing time** | < 2 seconds | For documents under 10,000 words |
| **Recommended max** | 50KB | Optimal performance for linting |

### API Request Validation

The `/api/lint` endpoint enforces strict validation (from `app/api/lint/route.ts:21-43`):

```typescript
// Text validation
if (!text || typeof text !== 'string') {
  return NextResponse.json({ error: 'No text provided' }, { status: 400 });
}

// Length check
if (text.length > MAX_TEXT_LENGTH) {
  return NextResponse.json({ error: 'Text too large' }, { status: 400 });
}

// Style guide whitelist
if (!ALLOWED_STYLES.includes(style)) {
  return NextResponse.json({ error: 'Invalid style guide' }, { status: 400 });
}

// Custom rule limits
```

These limits prevent abuse and ensure reliable performance.

### Security Measures

<Info>
  Verbalize implements several security protections to ensure safe operation:
</Info>

- **Style guide whitelist**: Only Google, Microsoft, and RedHat are allowed
- **File name sanitization**: Custom rule filenames cleaned of dangerous characters
- **Path traversal prevention**: Only base filenames accepted (no `../` paths)
- **Temporary file cleanup**: All temp files deleted after processing
- **No HTML rendering**: `react-markdown` does not render raw HTML by default

## Vale Configuration

### Dynamic Configuration Generation

Verbalize generates a `.vale.ini` file per lint request:

```ini
StylesPath = /tmp/vale-xyz/styles
MinAlertLevel = suggestion
[*.md]
BasedOnStyles = Google, Custom
```

**Key configuration choices**:

- **StylesPath**: Points to temporary directory with symlinked styles
- **MinAlertLevel**: Set to `suggestion` to show all severity levels
- **BasedOnStyles**: Combines selected base style + Custom (if rules uploaded)

### Permanent Configuration

The project includes a base `.vale.ini` for local development:

```ini
StylesPath = styles
MinAlertLevel = suggestion
Packages = Google, Microsoft, RedHat
[*.md]
BasedOnStyles = Vale, Google, Microsoft, RedHat
```

This configuration is NOT used during web linting—only for local CLI usage.

### Style Directories

The `styles/` directory contains three subdirectories:

```
styles/
├── Google/       # ~50 rule files
├── Microsoft/    # ~40 rule files
└── RedHat/       # ~30 rule files
```

Each contains `.yml` rule files and a `meta.json` descriptor. These are symlinked into temporary directories during linting to avoid copying overhead.
